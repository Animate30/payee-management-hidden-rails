1 View payees / recipients List, browse, and locate saved payees ▾
Aligned across all rails
  • Every rail maintains a persistent list of saved payees / recipients
  • Each list entry displays at minimum: payee name and one primary identifier
  • All lists support basic navigation — tap or click to open a detail view
Partial / inconsistent across rails
  • Visual pattern varies significantly: cards (Zelle), plain rows (ACH, RTP), form-style tables (Wire), mixed lists (Transfers, Bill Pay)
  • Active / inactive status shown only in Transfers and Bill Pay — absent elsewhere
  • Sort order, filtering, and search vary widely or are entirely absent
  • No unified search that spans multiple payment rails
  • Payee count and management are siloed per rail — the same recipient can appear as separate entries in Zelle, ACH, and Wire
Show rail-by-rail breakdown
Transfers
Nickname + bank name + masked account. Separate tabs for internal vs external accounts.
Partial
Bill Pay
Biller name + masked account + autopay indicator + next due date. Richest list view of any rail.
Partial
RTP
Contact name + bank identifier. Minimal context; no status signals. Real-time availability not surfaced on list.
Partial
Zelle
Contact directory view tied to device contacts. Shows phone or email. No bank details visible.
Partial
ACH
Template name or beneficiary name; routing info shown. Company ID and SEC code not visible to customers.
Partial
Wire
Beneficiary name + bank / SWIFT. Domestic and international beneficiaries split into separate lists.
Partial
Missing across all rails
  • No unified payee roster that spans rails — customers must navigate to each rail separately to find a specific person
  • No deduplication or linking of the same real-world recipient across rails
  • No "quick pay" or frequent payee shortcut at the roster level
  • No last-payment date or payment frequency visible on list entries

A single payee roster at the customer level — not one per rail. Customers find who they pay; the system decides how.

Shared foundation — unify these
  • One payee roster page surfaces all saved payees, regardless of rail
  • Each roster entry shows: name / nickname, relationship type (person, business, biller), and rail badges indicating which rails are configured for that payee
  • Shared search, sort, and filter apply across all rails from a single entry point
  • Contextual metadata at list level: last payment date, payment frequency, verification status
  • "Quick pay" shortcut available from list view for recently used payees
Intentional rail differences — preserve these
  • Zelle: entries remain phone / email keyed; bank details are never stored or displayed (Zelle network requirement)
  • Wire: domestic vs international split preserved in payee type indicator; SWIFT / BIC shown in detail, not list
  • Bill Pay: autopay indicator and next due date remain visible in list — customers rely on these for payment planning
  • RTP: real-time capability status badge shown per payee (some accounts cannot receive RTP — this is operationally relevant)
Design principle: The list view is a shared container. The content within each payee entry can carry rail-specific signals where those signals have real meaning to the customer's next decision.
2 Add payee / recipient Onboarding a new person or entity to pay ▾
Aligned across all rails
  • All rails require a destination identifier to create a payee (account number, phone, email, routing, SWIFT)
  • All rails require a name or nickname to save a payee
  • All rails confirm intent — at minimum an explicit "Save" or "Add" action before persisting
Partial / inconsistent across rails
  • Verification method differs significantly and has no shared UX language:
Transfers
Micro-deposit verification (1–3 business days). Customer confirms two small amounts.
Delayed verification
Bill Pay
No verification. Account number stored as-entered. Errors surface only at payment time.
No verification
RTP
No dedicated "add payee" flow. Routing + account entered at payment time. No payee saved.
No add flow
Zelle
Phone or email lookup against Zelle directory. Instant. Recipient must be enrolled.
Instant lookup
ACH
Routing + account form. Prenote / micro-deposit optional. Often skipped in practice.
Optional verification
Wire
Complex form: legal name, address, bank SWIFT, intermediary (if applicable). No real-time validation.
Manual, no validation
Missing across all rails
  • No shared "Add payee" entry point — customers must navigate to each rail separately and repeat name / contact info
  • No progressive disclosure: adding the same person to Zelle and ACH requires two completely separate flows with no linkage
  • No deduplication prompt: the system allows identical real-world recipients to be saved as separate records across rails
  • No payee profile created at setup — payee identity is owned by the rail, not by the customer relationship

One "Add payee" door. Rail configuration happens inside, as modules — not as separate doors the customer has to find.

Shared foundation — unify these
  • Single "Add payee" entry point at the customer level
  • Step 1: name, nickname, relationship type (person / business / biller) — shared across all rails
  • Step 2: choose which rails to configure for this payee (multi-select)
  • Rail configuration modules load sequentially; shared identity fields (name, address) are not re-entered for each rail
  • Deduplication: system checks if a payee with a matching name or identifier exists — "James Miller already exists in Zelle. Link this ACH account to the same payee profile?"
  • Result: a unified payee profile record with rail-specific sub-records attached
Intentional rail differences — preserve these
  • Wire: full compliance fields — legal name, physical address, bank SWIFT / BIC, purpose of payment — cannot be simplified; required by OFAC screening and Reg J
  • Zelle: directory lookup flow remains Zelle-network-specific; no bank details captured or stored; cannot be merged with ACH or Transfers add flow
  • Transfers (external): micro-deposit timeline is regulatory, not a UX choice; the delay must be communicated clearly but cannot be removed
  • RTP: no persistent payee saved today — future state should introduce an RTP payee record, but instant account validation at enrollment must remain in the flow
Design principle: The shared foundation captures identity. Rail-specific modules capture capability. Customers shouldn't have to re-prove who someone is just because they're adding a new way to pay them.
3 View / edit payee Accessing and modifying saved payee details ▾
Aligned across all rails
  • All rails display payee name and primary identifier on the detail screen
  • All rails provide a delete / remove action
Partial / inconsistent — edit capability by rail

Edit capability varies significantly with no consistent UI signal about what is editable vs locked:

Transfers
Nickname editable. Routing + account locked — must delete and re-add to change bank details.
Nickname only
Bill Pay
Account number with biller can be edited. Biller name is locked. Next payment date accessible.
Account editable
RTP
No detail or edit view. No payee record exists to edit. Must start over if recipient info changes.
View-only / none
Zelle
Phone / email view only. Token cannot be changed in app. No nickname or notes field.
View-only
ACH
Nickname editable. Routing / account sometimes locked after prenote or verification.
Partial edit
Wire
View-only in most implementations. Full delete + re-entry required to change any field.
View-only
Missing across all rails
  • No unified payee profile view — customer cannot see "all the ways I can pay James Miller" in one place
  • No edit history, audit trail, or "last modified" timestamp for any rail
  • No ability to add personal notes or tags to a payee
  • No clear UI explanation of why a field is locked (compliance? network rule? verification state?)
  • No verification status transparency — customers don't know whether a payee is verified, pending, or unverified

A payee profile is the canonical record. Rail connections are modules within it — each with their own edit rules, clearly labelled.

Shared foundation — unify these
  • Canonical payee profile page: name, nickname, notes, relationship type, date added, last payment date
  • All shared fields (name, nickname, notes) editable at the profile level — changes apply across all configured rails
  • Rail configurations displayed as distinct, labeled modules: "Zelle connection," "ACH account," "Wire beneficiary"
  • Consistent edit intent signal: pencil icon = editable; lock icon = locked by network or compliance rule, with tooltip explanation
  • Audit trail: added date, last modified, and verification status visible per rail module
  • Customer-facing language: "Locked — this field was set during verification and cannot be changed. To update, remove and re-add."
Intentional rail differences — preserve these
  • Zelle module: phone / email shown as view-only — Zelle token cannot be changed; this is a network constraint, not a design choice
  • Wire module: legal name and address locked post-verification — required for OFAC screening integrity; editing = new verification cycle
  • RTP module: real-time capability status indicator (eligible / ineligible) — this is a live account-level attribute, not editable
  • ACH module: prenote status and account verification date shown; routing / account editable only before first transaction or after reversal
Design principle: The distinction between "locked by us" and "locked by a network rule we don't control" is meaningful to customers and must be surfaced — not hidden behind a generic disabled state.
4 Group payees Organizing payees into categories or collections ▾
Partial — only Bill Pay supports meaningful grouping
  • Bill Pay: category grouping supported (e.g., Utilities, Subscriptions, Mortgage). User-assigned or auto-categorized by biller type. Useful but scoped to Bill Pay only.
  • Transfers (some implementations): nickname-based organization can serve as a weak proxy for grouping, but no explicit group structure exists
Missing — no grouping in five of six rails
  • RTP, Zelle, ACH, and Wire: no grouping capability whatsoever — payees are an undifferentiated list
  • No cross-rail grouping: a customer cannot tag "family members" in a way that surfaces across Zelle and ACH simultaneously
  • No user-defined labels, tags, or favorites beyond Bill Pay's category system
  • No smart grouping based on payment behavior (e.g., "most paid," "recurring," "inactive")

Groups live at the payee profile layer — they're customer-defined metadata, not a rail feature. Every rail benefits automatically.

Shared foundation — unify these
  • User-defined groups at the payee profile layer, independent of rail — any payee can belong to one or more groups
  • Groups are metadata only — no impact on payment routing, limits, or execution logic
  • Default system groups to bootstrap organization: "Frequent," "Family," "Utilities," "Vendors"
  • Smart groups suggested based on payment history: "You've paid these 4 people every month — create a group?"
  • Groups appear as filter options on the payee roster and on payment history views
  • Favorites / starred payees as a lightweight grouping primitive available to all rail types
Intentional rail differences — preserve these
  • Bill Pay: biller categories (Utilities, Subscriptions) maintained as a separate, richer taxonomy — these are biller-type categories, not customer-defined groups, and carry different meaning
  • Rail-type tags (e.g., "Zelle contact," "Wire beneficiary") are system-generated descriptors, distinct from customer-defined groups
Design principle: Customer-defined groups and system-defined rail or biller taxonomy coexist. They solve different problems — don't merge them. A "Utilities" biller category is objective; a "Family" group is personal.
5 Manage / settings Per-payee controls, notifications, and deletion ▾
Aligned across all rails
  • All rails offer a delete / remove action for saved payees (though confirmation UX and consequence clarity vary)
Partial / inconsistent across rails
  • Nickname editing: available in Transfers and ACH; absent in Zelle, RTP, and Wire
  • Notification settings: Bill Pay (payment alerts, autopay reminders per biller); Zelle (at network level only, not per-payee); all other rails — none
  • Verification status: visible in Transfers (micro-deposit pending / complete); absent in all other rails
Show settings availability by rail
Transfers
Nickname edit, delete, micro-deposit status. No notification controls per payee.
Bill Pay
Autopay toggle, payment alerts, account edit, delete. Most complete settings of any rail.
RTP
No payee settings surface. Add / remove only. No notifications or status.
Zelle
Delete only. Notification preferences exist at the Zelle level, not per contact.
ACH
Nickname edit, delete, prenote status. No notification controls.
Wire
Delete only. Beneficiary verification / hold status not exposed to customers.
Missing across all rails
  • No shared settings layer: nickname, notes, and notifications must be configured per rail if available at all
  • Deleting a payee in one rail has no effect on other rails and produces no prompt to manage the cross-rail relationship
  • No explanation of deletion consequences: customers don't know whether removing a Zelle contact also removes their Zelle token registration
  • No activity log or payment history accessible from payee settings
  • No ability to "pause" a payee without deleting them

Settings that belong to the payee relationship are shared. Settings that belong to the rail are owned by the rail module — but visible in one place.

Shared foundation — unify these
  • Shared settings panel per payee profile: nickname, notes, payment notifications on / off, mark as favorite
  • Activity summary in settings: last payment date, total payments in last 12 months, first payment date
  • "Danger zone" section with scoped deletion UI: "Remove from all rails" or "Remove only from Wire" — customer chooses scope
  • Deletion consequence language required before confirmation: "Removing from Zelle disconnects the Zelle payment token. Your Zelle account remains active and this contact can be re-added."
  • Pause / archive option: deactivate a payee without deletion — visible in "Archived" filter, restorable
Intentional rail differences — preserve these
  • Bill Pay: autopay settings and payment schedule remain within the Bill Pay module — these are biller-relationship settings, not payee-profile settings
  • Wire: compliance hold status and OFAC screening status are operational fields managed at the rail level; not customer-editable, but should be visible
  • ACH: prenote status, company ID filter, and SEC code type remain ACH-module-specific settings
  • RTP: real-time capability status (eligible / ineligible) is a live attribute, not a setting — shown as informational only
Design principle: The shared settings panel is for things the customer controls about the relationship. Rail modules surface things the system controls about the capability. The customer should see both — but the hierarchy must be clear.
6 Education / safety Fraud awareness, risk signals, and customer guidance ▾
Aligned across all rails
  • None — there is no safety or education layer applied consistently across all rails today
Partial — exists only in Zelle and Wire
  • Zelle: strong, Zelle-network-mandated fraud warnings present throughout the add and send flows — "Only send money to people you know and trust." Positioned at point of transaction, not payee setup.
  • Wire: irreversibility notice shown at payment execution — "Wire transfers cannot be cancelled or reversed once sent." Not present at payee setup.
  • Bill Pay: some billers surface account verification guidance during add — inconsistent, biller-driven, not system-driven.
Missing — critical gaps
  • Transfers, RTP, and ACH: zero proactive safety or fraud education during payee setup or management
  • No consistent risk level indicator to help customers assess a new payee (new vs known, verified vs unverified)
  • Safety warnings appear only at payment execution — by then, the payee is already "trusted" by the system. The teachable moment is payee creation.
  • No stale payee prompts: no system check for payees not paid in 6–12+ months with a "confirm details before sending" reminder
  • No customer-facing explanation of what happens if a payment goes to the wrong account — each rail has a different answer (reversal window, finality, dispute process)
  • No unified language for "what kind of account is this" — risk context (new wire beneficiary vs known biller) is not communicated

Safety education moves to the moment of payee creation — before trust is established, not after a mistake is made.

Shared foundation — unify these
  • Safety layer applied at payee creation for all rails — brief, contextual, not alarming for low-risk scenarios
  • Risk level indicator per payee type, displayed at profile level:
    · Low risk — verified biller, known institution, established relationship
    · Medium — unverified individual, first-time recipient, no prior payment history
    · Higher risk — new wire beneficiary, international, no prior relationship
  • Stale payee prompt: "You haven't paid [Name] in 180 days. Confirm their details are still correct before sending."
  • Rail-appropriate recovery language at setup — not buried post-failure: "If this payment goes to the wrong account, here's what we can do."
  • Consistent copy principles: specific, honest, and actionable — not generic legal disclosures customers skip
Intentional rail differences — preserve these
  • Wire: mandatory irreversibility disclosure required at both payee setup AND payment execution — this is a compliance obligation under Reg J and UCC Article 4A; cannot be unified away or reduced to a shared generic warning
  • Zelle: Zelle-network fraud messaging must be retained verbatim as a network participation requirement; Truist cannot modify or replace this copy
  • RTP: real-time finality disclosure retained — payments are irrevocable in practice even if technically different from wire; customers need to know this is different from ACH
  • ACH: return and reversal window disclosure is unique to ACH and valuable — customers need to understand this is the one rail with a meaningful error-recovery window (typically 2 banking days)
Design principle: The shared safety layer raises the floor — every rail gets baseline education. Rail-specific disclosures raise the ceiling where compliance or network rules require it. The shared layer must never water down or replace a mandated disclosure.
Design principles for the unified future state
Layered consolidation, not forced uniformity
A shared payee foundation sits above rail-specific modules. Consolidation is real at the customer layer. Rail differences are preserved below it — not papered over.
Identity once, capability as needed
A customer should prove who a payee is once. Configuring a new payment rail for that payee should never require re-establishing identity from scratch.
Explain the lock, not just the lock
When a field cannot be edited, the UI must say why — and whether the constraint is a product decision, a network rule, or a compliance requirement. Customers deserve the distinction.
Safety at setup, not just at send
Risk education and fraud awareness belong at payee creation — the moment when trust is being established. Point-of-payment warnings arrive too late to change the customer's mental model.
